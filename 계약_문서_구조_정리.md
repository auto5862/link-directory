# 계약 문서 구조 — 10개 vs 1개

---

## 그 사람 방식 (10장 문서)

| # | 계약 문서 | 배포 | 등록 |
|---|-----------|------|------|
| 1 | 문서 1 (.sol) | 배포 → 주소1 | 소스 검증, Record Token, API, 토큰 평판... |
| 2 | 문서 2 (.sol) | 배포 → 주소2 | 소스 검증, Record Token, API, 토큰 평판... |
| ... | ... | ... | ... |
| 10 | 문서 10 (.sol) | 배포 → 주소10 | 소스 검증, Record Token, API, 토큰 평판... |

→ **계약 1개 = 문서 1개 = 배포 1번 = 등록 1세트**

---

## 배포 전 / 배포 후에 계약서 넣는 시점

| 시점 | 할 일 |
|------|------|
| **배포 전** | 각 .sol 파일 준비 → TronIDE에 넣고 Deploy |
| **배포 후** | **소스 검증** 시 해당 .sol 파일 **업로드** (계약마다 자기 파일) |

→ 배포 전: 코드 준비  
→ 배포 후: 소스 검증에 **그 계약에 맞는 .sol 파일** 업로드

---

## 우리가 잘못 이해한 점

| 잘못 | 맞음 |
|------|------|
| 문서 1개로 토큰 업데이트 전부 처리 | **계약 1개 = 문서 1개 = 등록 1세트** |
| 한 번에 다 해결 | 각 계약마다 **따로** 소스 검증, Record Token, API 등록 |

---

## 우리 현재 (Link USD)

| 계약 | 문서 | 배포 | 등록 |
|------|------|------|------|
| Link USD (메인넷) | TetherUSD_Mainnet_Verify.sol | TNXDJX... ✓ | 소스 검증 ✓, 토큰 평판 대기 |
| Link USD (Nile) | LinkUSD_TRON.sol | TBBsTXP5... ✓ | 소스 검증 ✓ |

→ 우리는 **토큰 1개**라서 문서 1개로 진행한 게 맞음.  
→ **토큰이 10개**면 문서 10개, 등록 10세트 필요.

---

## 10개 문서 있을 때 정리 방법

| 순서 | 할 일 |
|------|------|
| 1 | 10개 .sol 파일 각각 확인 |
| 2 | 필요한 계약만 골라서 배포 |
| 3 | **배포한 계약마다** 소스 검증 (해당 .sol 업로드) |
| 4 | **배포한 계약마다** Record Token, API, 토큰 평판 등록 |

→ **계약 1개당 등록 1세트.** 한 문서로 전부 처리하는 게 아님.
